## 📌 一句话总结

**MTGR 是美团提出的工业级生成式推荐框架**，通过用户样本聚合、交叉特征保留、改进 HSTU 架构，在保持训练成本不变的情况下实现单样本前向推理 FLOPs 提升 65 倍，在美团外卖核心场景落地，离线 CTCVR GAUC +2.88pp，订单量 +1.22%、PV_CTR +1.31%。

## ❓ 问题背景

### 2.1 扩展性两难选择

1. **DLRM 的瓶颈** - 依赖大量手工设计的交叉特征，计算复杂度随候选集线性增长，无法利用大模型并行训练优势
2. **生成式推荐的缺陷** - 借鉴 NLP 的 Next Token Prediction 范式，但需丢弃交叉特征，精度下降无法通过 scaling up 弥补

### 2.2 核心挑战

1. **如何保留交叉特征** - 生成式推荐需放弃传统 DLRM 精心构建的交叉特征，导致性能下降
2. **如何降低推理成本** - 候选集数量增长导致计算量线性增长
3. **如何提升训练效率** - 用户行为序列增长导致训练成本过高

## 💡 方法框架

### 3.1 核心贡献

1. **用户样本聚合** - 将同一用户的 K 个候选聚合为单条样本，减少冗余计算
2. **交叉特征保留** - 将交叉特征整合进候选 Token，保留 DLRM 优势
3. **改进的 HSTU** - 引入 Group-Layer Normalization 和 Dynamic Masking Strategy
4. **推理优化** - 动态 Batch Size、TensorRT 推理框架、Cutlass-based HSTU Kernel

### 3.2 数据重排

**DLRM 模式：** 一个用户的 K 个曝光行为生成 K 条训练样本，每条样本重复包含用户 Profile、行为序列等信息

**MTGR 模式：** 将同一用户在特定时间段内的所有候选物品聚合为单条样本

```
D = [U, S⃗, R⃗, [C₁,I₁], [C₂,I₂], ..., [Cₖ,Iₖ]]
```

**优势：**
- 训练效率：样本量从"曝光次数"降为"用户请求次数"
- 推理友好：一次前向计算得到所有 K 个候选物品的分数

### 3.3 特征 Token 化

| 特征类型 | Token 化方式 |
|----------|--------------|
| 用户画像 (U) | 每个标量特征变独立 Token |
| 行为序列 (S, R) | 每个历史行为经 MLP 映射为一个 Token |
| 候选集与交叉特征 | 候选 Item 特征与交叉特征拼接后经 MLP 映射为 Token |

### 3.4 改进的 HSTU

**Group-Layer Normalization (GLN)** - 按领域分组对不同领域的 Token 进行归一化，确保每个领域内的 Token 分布相似

**Dynamic Masking Strategy** - 根据 Token 的类型和时间关系，灵活控制哪些 Token 可以"看到"其他 Token 的信息，避免信息泄露

### 3.5 推理优化

1. **Cutlass-based HSTU Kernel** - 支持变长序列无需 padding，单算子性能提升 2-3 倍
2. **动态 Batch Size** - 每张卡的 BS 根据实际数据序列长度动态调整
3. **TensorRT 推理框架** - 算子融合、低精度量化能力

## 🔍 对比与思考

### 与 Meta GRs 对比

| 维度 | Meta GRs | MTGR |
|------|----------|------|
| 交叉特征 | 丢弃 | 保留 |
| 架构基础 | HSTU | 改进 HSTU |
| 优化重点 | 模型架构 | 工程优化 |

### 核心洞察

1. **交叉特征的价值** - 在工业推荐场景中，精心设计的交叉特征仍有不可替代的价值
2. **用户级聚合** - 从"样本为中心"到"用户为中心"的范式转变
3. **工程优化的必要性** - 理论架构需配合工程优化才能落地
