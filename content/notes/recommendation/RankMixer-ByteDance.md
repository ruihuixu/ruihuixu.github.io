## 📌 一句话总结

**RankMixer 是字节跳动提出的硬件感知精排架构**，通过 Token Mixing + Per-Token FFN 设计，将 GPU 利用率 (MFU) 从 4.5% 提升至 45%，实现 70 倍参数扩展（16M → 1.1B）而推理延迟保持不变，在抖音主站带来 APP 使用时长 +1.08% 的显著收益。

## ❓ 问题背景

### 2.1 工业推荐系统扩展的两大障碍

1. **严格的延迟和 QPS 约束** - 端到端延迟 <50ms，精排阶段 <15ms，支持百万级 QPS，无法像 LLM 那样自由扩展参数
2. **CPU 时代架构的低 GPU 利用率** - 现有工业推荐系统的特征交叉模块大多继承自 CPU 时代，核心算子是 Memory-bound 而非 Compute-bound，MFU 通常只有 4.5% 左右

### 2.2 传统特征交叉模块的三大问题

1. **内存瓶颈问题** - 大量嵌入查找和小型矩阵运算导致 GPU 利用率低下
2. **异构特征空间问题** - 数百个特征字段的语义空间差异很大，基于内积相似性的自注意力机制难以有效学习跨特征空间的关系
3. **高频率特征主导问题** - 所有 Token 共享同一组 FFN 参数，导致高频特征主导模型学习，长尾特征的信号被淹没

## 💡 方法框架

### 3.1 核心贡献

1. **Token Mixing 机制** - 提出无参数的跨 Token 信息交换方式，替代传统自注意力，在保持性能的同时显著提升计算效率
2. **Per-Token FFN 设计** - 为每个 Token 分配独立参数，实现特征子空间隔离建模，解决高频特征主导问题
3. **DTSI-MoE 扩展方案** - 采用 Dense 训练、Sparse 推理的策略，支持未来 10B 级别参数扩展
4. **Scaling Law 验证** - 在万亿级数据上验证了推荐系统的参数 - 性能幂律关系，从 16M 扩展到 1.1B 参数持续获得收益

### 3.2 总体架构

```
输入特征 → Tokenization → [Token Mixing + Per-Token FFN] × L → Mean Pooling → 预测
```

每个 RankMixer Block 包含两个核心组件：
1. **Multi-Head Token Mixing 模块** - 无参数的跨 Token 信息交换
2. **Per-Token FFN 模块** - 独立参数的特征子空间建模

可选地，PFFN 可以替换为 Sparse-MoE 变体以支持更大规模扩展。

### 3.3 特征 Tokenization

**核心挑战：** Token 数量需要在两个极端之间找到平衡

- Token 太多（数百个）→ 每个 Token 的参数和计算量过少，GPU 利用率不足
- Token 太少（1 个）→ 模型退化为简单 DNN，特征空间容易被主导

**解决方案：基于语义的特征分组**
- 将特征按语义聚类（用户画像、视频特征、序列特征、交叉特征等）
- 将组内特征拼接成统一向量，再分组切分为固定维度的 Token
- 通过投影层得到最终表示

### 3.4 Multi-Head Token Mixing

**核心思想：** 用无参数操作替代自注意力，实现跨 Token 信息交换

**机制详解：**
1. **头部分割** - 将每个 Token 均匀切分为 H 个头
2. **维度重组** - 按头维度将不同 Token 的对应头拼接在一起，实现信息混合
3. **输出** - 保持 Token 数量不变以便残差连接，通常设置 H = T

**为什么不用自注意力：**

| 维度 | 自注意力 | Token Mixing |
|------|----------|--------------|
| 计算复杂度 | O(T²) | O(T) |
| 参数量 | 需要存储注意力权重矩阵 | 无参数 |
| 异构特征适应性 | 假设统一嵌入空间 | 适合异构特征 |
| AUC | -0.03% | 基准 |
| FLOPs | +71.8% | 基准 |

### 3.5 Per-Token FFN

**核心思想：** 为每个 Token 分配独立参数，解决特征子空间主导问题

**设计：** 对第 t 个 Token 使用独立的 FFN 参数
```
v_t = fᵗ_pffn,₂(Gelu(fᵗ_pffn,₁(s_t)))
```

**与相关架构的区别：**

| 架构 | 参数共享策略 | 应用场景 |
|------|--------------|----------|
| MMoE | 多专家共享同一输入 | 多任务学习 |
| Transformer FFN | 不同 Token 共享同一组参数 | 通用序列建模 |
| RankMixer PFFN | 不同输入加不同参数 | 异构特征建模 |

### 3.6 Sparse-MoE 扩展方案

**两大问题与解决方案：**

**问题 1：均匀 k-expert 路由**
- 传统 Top-K 选择对所有 Token 一视同仁，导致低信息 Token 浪费预算，高信息 Token 饥饿
- **解决方案：** 采用 ReLU Routing，用 ReLU 激活替代 Top-K+Softmax，使高信息 Token 激活更多专家

**问题 2：专家训练不足**
- Per-Token FFN 已使专家数爆炸，路由不平衡导致专家训练不充分，出现死亡专家现象
- **解决方案：** 采用 DTSI（Dense-Training / Sparse-Inference）策略
  - 训练阶段：使用训练路由器，不应用正则化损失，保证专家充分训练
  - 推理阶段：使用推理路由器，应用正则化控制激活专家数

**效果：** 在 1/8 稀疏度下，采用 DTSI+ReLU Routing 后仅下降 0.08%，几乎保持 1B Dense 模型的全部精度，同时推理吞吐量提升 50%

### 3.7 Scale Up

**扩展的四个正交轴：** Token 数量 T、模型宽度 D、层数 L、专家数量 E

**Scaling Law 发现：**
- 模型质量主要与总参数量相关，不同扩展方向（L、D、T）的性能几乎相同
- 从计算效率角度，增大 D 比增加 L 更优，因为更大矩阵乘法带来更高 MFU
- 推荐配置：L = 2，采用浅而宽的架构

## 🔍 对比与思考

### 与传统 DLRM 对比

| 维度 | DLRM | RankMixer |
|------|------|-----------|
| 计算模式 | Memory-bound | Compute-bound |

### 核心洞察

1. **从 Memory-bound 到 Compute-bound** - 推荐模型设计需从 CPU 时代思维转向 GPU 时代思维
2. **Per-Token 设计** - 独立参数分配是解决异构特征主导问题的关键
3. **Scaling Law 成立** - 推荐系统在万亿级数据上验证参数 - 性能幂律关系
