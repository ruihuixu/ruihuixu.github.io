## 📌 一句话总结

**OneTrans 是字节跳动提出的统一 Transformer 推荐架构**，通过将序列特征和非序列特征统一转换为 token 序列，采用混合参数化 OneTrans Block 和金字塔堆叠策略，实现双向信息交换和高效计算，在在线 A/B 测试中带来人均 GMV +5.68% 的显著收益。

## ❓ 问题背景

### 2.1 传统推荐架构的两大痛点

工业推荐的精排传统上拆成两条线各自"做大"：

1. **序列建模** - 把用户多行为长序列喂给 DIN / Transformer / LONGER
2. **特征交互** - 把用户、物品、上下文等非序列特征堆到 DCNv2 / Wukong / RankMixer 等交叉模块里

这种 encode-then-interaction 流水线有两大痛点：

1. **序列与静态/上下文特征之间的信息流动是"单向后融合"** - 非序列特征难以影响序列建模的兴趣表示，限制了表达力
2. **工程上被拆成两段** - 难以继承 LLM 体系的高效优化（KV 缓存、FlashAttention、混精等）

### 2.2 模块分离的局限性

| 问题 | 描述 |
|------|------|
| 双向信息交换受限 | static/context 的非序列特征难以影响序列建模的兴趣表示 |
| 工程扩展困难 | 模块独立增加推理时延，难以利用 LLM 高效优化技术 |

## 💡 方法框架

### 3.1 核心贡献

1. **统一 Tokenizer** - 把多行为序列与非序列属性都映射成 token 序列（序列在前，非序列在后）
2. **混合参数化 (Mixed Parameterization)** - 所有 S-tokens(序列) 共享一套 Q/K/V 与 FFN；而每个 NS-token(非序列) 拥有自己的 token-specific Q/K/V 与 FFN
3. **因果注意力 + KV 缓存** - 既保住"时序因果"，又能跨候选/跨请求缓存中间结果来降本
4. **金字塔堆叠策略** - 逐步修剪序列 token，提升计算效率

### 3.2 总体架构

```
统一 Tokenizer → [OneTrans Block] × L → 金字塔压缩 → 输出
```

传统架构 vs OneTrans 架构：

| 架构 | 流程 | 问题 |
|------|------|------|
| 传统 | 序列建模 → 向量压缩 → 与非序列特征拼接 → 特征交互块 | 信息单向流动，模块碎片化 |
| OneTrans | 序列 + 非序列统一 token 化 → 联合建模 | 双向信息交换，工程优化友好 |

### 3.3 统一 Tokenizer

**非序列特征 Token 化（两种方式）：**

| 方式 | 描述 |
|------|------|
| Group-wise Tokenizer | 特征先按照语义分组（context/doc/user 侧等）然后分别过 MLP 映射到相同维度 d |
| Auto-Split Tokenizer | 所有特征先拼接起来过 MLP 然后再 split，类似 RankMixer 做法 |

**序列特征 Token 化：**

首先多条序列拼接各自的 side info 后各自过 MLP 映射到相同维度 d，然后有两种合并方式：

1. **按时间片合并（推荐）** - 把所有序列的 item 统一按照序列 item 的时间片先后顺序合并，并且引入序列类型的信号，把最新的序列 item 放在后面
2. **按语义合并** - 序列按照语义合并（如点击序列→加入购物车序列→购买序列），把高重要性的序列放在前面，序列之间加上 [SEP]

### 3.4 OneTrans Block

**设计动机：** 考虑序列和非序列特征的数值范围与统计特性存在显著差异

**核心组件：**

1. **RMSNorm** - 将数值拉到同等范围（预归一化）
2. **MixedMHA（混合多头注意力）** - 所有序列 tokens 共享 qkv 参数，不同的非序列 token 之间各自具有不同的 qkv 参数
3. **MixedFFN（混合前馈网络）** - 类似 MixedMHA 的参数化策略
4. **因果注意力掩码** - 确保非序列 token 可访问整个序列历史，但反过来不行

**注意力计算：**
```
Attention(Q, K, V) = Softmax(QK^T / sqrt(d))V
```

其中 Q/K/V 按位置分配：S-tokens 共享权重，NS-tokens 使用独立权重。

### 3.5 金字塔堆叠策略

随着网络层数加深，仅保留最近的部分 S-tokens 作为查询，逐步压缩序列长度。

**复杂度优化：** 从 O(T²) 降至 O(T_query · T)

### 3.6 训练与部署优化

**跨请求 KV 缓存：**
- 用户行为序列的中间结果被缓存并跨候选物品重用
- 计算复杂度从 O(N·T²) 降至 O(T² + N·T)

**其他优化：**
- FlashAttention 集成
- 混合精度训练
- 激活重计算

## 🔍 对比与思考

### 与 RankMixer 对比

| 维度 | RankMixer | OneTrans |
|------|-----------|----------|
| 统一方式 | 将序列特征视作非序列特征 | 将非序列特征视作序列特征的 target |
| 堆叠模块 | 特征交叉模块 | Attention 模块 |
| 工程友好 | 直接干掉 attention，易扩展 | 容易迁移到其他业务 |
| 风格 | 务实 | 务实，不蹭 GR 概念 |

### 与 Homer 对比

| 维度 | Homer | OneTrans |
|------|-------|----------|
| 特征融合 | 在 target 和序列里都加入非序列特征 | 统一 token 化后联合建模 |
| 特色 | 针对特征异质性专门处理 | 混合参数化 + 因果注意力 |

### 核心洞察

1. **统一序列和非序列特征是共识** - 对硬件友好，更容易扩展参数
2. **"分久必合"趋势** - 模块独立增加推理时延，统一架构是方向
3. **字节的务实风格** - 不硬蹭生成式概念，注重实际业务收益
4. **双向信息交换价值** - 非序列特征可以前向影响序列表征

### 本质

OneTrans 的核心价值是拆掉"模块墙"，用一个统一的 Transformer 主干，把"用户多行为序列 + 非序列特征"统一进一段 token 流里，一次性完成"序列建模 + 特征交互"，既释放双向信息交换，又吃到 LLM 的全家桶优化。
