## 📌 一句话总结

**OneREC 是快手提出的首个统一"召回 + 精排"的端到端生成式推荐模型**，采用 Encoder-Decoder 架构、稀疏 MoE 扩展容量、引入 DPO 直接偏好优化，在快手主站核心场景带来用户观看时长 +1.6% 的提升，服务成本降至传统方案的 1/10。

## ❓ 问题背景

### 2.1 传统级联架构的痛点

1. **碎片化计算** - 超过一半的算力用于搬运和存储数据，模型运算效率低
2. **优化目标不一致** - 召回、粗排、精排各阶段目标函数不一致，导致信息传递损耗
3. **框架落后** - 级联架构制约推荐系统吸纳最新 AI 技术，与 LLM 技术发展存在代差

### 2.2 生成式推荐的困境

1. **召回与精排割裂** - 现有生成式模型多局限于召回阶段，后续仍需接入复杂精排模型
2. **推理延迟与生成质量平衡** - 亿级用户/物品规模下难以兼顾
3. **偏好对齐缺失** - 缺乏与用户真实偏好的直接对齐机制

## 💡 方法框架

### 3.1 核心贡献

1. **端到端架构** - 统一召回 + 精排，采用 Encoder-Decoder 架构
2. **稀疏 MoE** - 扩展模型容量，支持更大规模参数
3. **DPO 偏好对齐** - 首次将直接偏好优化引入推荐系统
4. **迭代式偏好对齐 (IPA)** - 多轮偏好优化提升生成质量

### 3.2 整体架构

OneREC 采用 Encoder-Decoder 架构，将推荐问题转化为序列生成任务，由四个核心部分组成：

```
分词器 → 编码器 → 解码器 → 奖励系统 (微调)
```

### 3.3 分词器 (Tokenizer)

**挑战：** 生成式推荐面对十亿级规模物品空间，直接生成原子级 ID 面临巨大计算和结构挑战

**解决方案：** 将物品编码为由粗到细的语义 ID，使用缩减且固定的词表

**多模态融合：**
- 视频的标题、标签、语音转文字、图像识别等多维信息
- 融入用户行为信息（协同信号）

**分层语义编码：** 采用 RQ-Kmeans 技术，将每个视频转化为 3 层粗到细的语义 ID

### 3.4 编码器 (Encoder)

**特征工程 - 序列构建：**

| 序列类型 | 内容 |
|----------|------|
| 用户静态路径 | 用户的静态属性信息 |
| 短期路径 | 近 20 条用户曝光记录 |
| 正反馈路径 | 近 256 条互动行为记录 |
| 终身路径 | 对原始 2000 视频序列进行 2 层分层压缩，每层压缩到 128 长度 |

### 3.5 解码器 (Decoder)

采用 Point-wise 输出 Token，整体框架类似 Transformer 的 Decoder，融合 MoE 以及 Top-K 路由

**Loss：** 基于语义 ID 的 Next Token Prediction

### 3.6 偏好对齐

**两阶段训练：**
1. **第一阶段** - Next-Token-Prediction 任务
2. **第二阶段** - 采用 DPO 进行偏好对齐

**奖励模型：** 设计奖励模型模拟用户生成行为，根据推荐系统在线学习特性定制采样策略

## 🔍 对比与思考

### 与 OneRec-V2 对比

| 维度 | OneREC v1 | OneRec-V2 |
|------|-----------|-----------|
| 架构 | Encoder-Decoder | Lazy Decoder-Only |
| 计算效率 | 97.66% 资源用于编码 | 计算量减少 94% |
| 训练资源 | - | 减少 90% |

### 核心洞察

1. **端到端价值** - 统一召回 + 精排，消除级联不一致
2. **DPO 在推荐的适用性** - 直接偏好优化可有效对齐用户兴趣
3. **稀疏 MoE 扩展** - 在不增加推理成本的前提下扩展模型容量
