## ⚠️ 高频错误（先看这里）

### ❌ 错误 1：55 题用 `i >= len(nums)` 判断越界而非比较 `max_reach`

```python
# 错误：for 循环本身保证 i < len(nums)，这个判断永远不会触发
if i >= len(nums):
    return False
```

```python
# 正确：判断当前位置是否超出「已确认可达范围」
if i > max_reach:
    return False
```

**原因**：`max_reach` 是贪心维护的最远可达下标，与 `len(nums)` 无关。应比较遍历位置 `i` 与 `max_reach`，而非与数组长度比较。`i > max_reach` 意味着从起点无法到达 `i`，此时必须提前返回 False。

---

### ❌ 错误 2：45 题在到达边界之前就推进 `cur_end`（更新时机错误）

```python
# 错误：先判断边界再更新 next_end，导致边界处收集的范围不完整
for i in range(len(nums) - 1):
    if i == cur_end:
        jumps += 1
        cur_end = next_end        # BUG：此时 i 处的 next_end 还没被计入
    next_end = max(next_end, i + nums[i])
```

```python
# 正确：先更新 next_end，再在边界处推进 cur_end
for i in range(len(nums) - 1):
    next_end = max(next_end, i + nums[i])
    if i == cur_end:
        jumps += 1
        cur_end = next_end
```

**原因**：在走到 `cur_end` 之前，需要持续收集当前这一跳覆盖范围内所有位置能到达的最远距离，收集完毕后才推进边界。顺序颠倒会导致 `cur_end` 处自身的贡献被漏掉。

---

### ❌ 错误 3：134 题起点更新逻辑错误——在循环内提前返回

```python
# 错误：tank < 0 时直接 return i，或 return i+1
if tank < 0:
    return i + 1   # BUG：缺少全局 total 的有解性验证
```

```python
# 正确：重置 start = i+1 和 tank，遍历完后凭 total >= 0 决定是否有解
if tank < 0:
    start = i + 1
    tank = 0
return start if total >= 0 else -1
```

**原因**：`tank < 0` 只说明「从当前候选区间任一点出发都不够用」，候选起点右移到 `i+1`。但能否完成一圈，必须等全局 `total >= 0` 验证后才能确定，不能在循环内提前返回。

---

### ❌ 错误 4：135 题只做一次从左到右的扫描，忽略右侧约束

```python
# 错误：只满足了「右边评分高则糖果多」，没有处理「左边评分高则糖果多」
for i in range(1, n):
    if ratings[i] > ratings[i - 1]:
        candy[i] = candy[i - 1] + 1
return sum(candy)   # BUG：遗漏从右向左的约束
```

```python
# 正确：第二次从右到左，取 max 保留两侧约束中的较大值
for i in range(n - 2, -1, -1):
    if ratings[i] > ratings[i + 1]:
        candy[i] = max(candy[i], candy[i + 1] + 1)
```

**原因**：从左到右只能保证「右侧约束」；从右到左才能捕获「左侧约束」。对同一位置，两个方向各自独立得出的最小值，取 `max` 后才能同时满足两侧约束。

---

### ❌ 错误 5：860 题收到 20 美元时优先用 $5×3 而非 $10+$5

```python
# 错误：先消耗三张 5 元，浪费了「万能零钱」
if five >= 3:
    five -= 3
elif ten > 0 and five > 0:
    ten -= 1; five -= 1
```

**原因**：5 元是「万能零钱」，既能找 10 元账单（唯一方案），又能凑 20 元账单的找零；而 10 元只能用于找 20 元账单。收到 20 时优先消耗用途较窄的 10 元，保留通用性更强的 5 元，以应对后续 10 元账单的找零需求。

## 详解

### 55. 跳跃游戏

#### 核心思路

不枚举每种跳法，只维护「当前能到达的最远位置」`max_reach`。

遍历每个下标 `i` 时：先判断 `i` 是否仍在可达范围内（`i > max_reach` 则不可达，提前返回 False）；再用 `i + nums[i]` 更新 `max_reach`；若 `max_reach >= len(nums) - 1`，终点可达，返回 True。

**贪心正确性**：只要 `i` 可达，从 `i` 出发可到达 `[i, i + nums[i]]` 内任意位置；对 `max_reach` 取 max 意味着我们始终掌握最大覆盖范围，不存在「跳得更少反而更优」的情况。

#### Python 代码

```python
def canJump(nums: list[int]) -> bool:
    max_reach = 0                        # 当前能到达的最远下标
    for i in range(len(nums)):
        if i > max_reach:                # i 已超出可达范围，无法继续
            return False
        max_reach = max(max_reach, i + nums[i])
    return True
```

#### 图解

```text
nums = [2, 3, 1, 1, 4]
下标:    0  1  2  3  4

i=0: max_reach = max(0, 0+2) = 2
i=1: max_reach = max(2, 1+3) = 4  →  4 >= 4（最后下标），可达，返回 True ✓

nums = [3, 2, 1, 0, 4]
下标:   0  1  2  3  4

i=0: max_reach = max(0, 0+3) = 3
i=1: max_reach = max(3, 1+2) = 3
i=2: max_reach = max(3, 2+1) = 3
i=3: max_reach = max(3, 3+0) = 3
i=4: 4 > max_reach=3 → 返回 False ✓
```

### 45. 跳跃游戏 II

#### 核心思路

在 55 基础上，问最少跳跃次数。维护两个变量：
- `cur_end`：当前这一跳的右边界（走到这里必须再跳）。
- `next_end`：在当前跳覆盖的范围内，能到达的最远位置（下一跳候选上界）。

遍历时不断更新 `next_end`；当 `i == cur_end` 时，`jumps++`，将 `cur_end` 推进到 `next_end`。

只遍历到 `n-2`：到达倒数第一个位置时已在终点，无需再计步；若遍历至 `n-1` 并在 `i == cur_end` 处触发，会多加一步。

#### Python 代码

```python
def jump(nums: list[int]) -> int:
    cur_end = 0    # 当前跳能到达的边界
    next_end = 0   # 下一跳最远候选位置
    jumps = 0

    for i in range(len(nums) - 1):   # 只遍历到 n-2
        next_end = max(next_end, i + nums[i])
        if i == cur_end:             # 走到边界，必须再跳一次
            jumps += 1
            cur_end = next_end       # 边界推进到下一跳的最远处

    return jumps
```

#### 图解

```text
nums = [2, 3, 1, 1, 4]
下标:    0  1  2  3  4

初始：cur_end=0, next_end=0, jumps=0

i=0: next_end = max(0, 0+2) = 2
     i==cur_end(0) → jumps=1, cur_end=2

i=1: next_end = max(2, 1+3) = 4
i=2: next_end = max(4, 2+1) = 4
     i==cur_end(2) → jumps=2, cur_end=4

i=3: next_end = max(4, 3+1) = 4
     i(3) ≠ cur_end(4)，不加步

结果：jumps = 2 ✓

区间接力示意（下标 0~4）：
  第 1 跳覆盖范围：[0, 2]
  第 2 跳覆盖范围：[1, 4]  ← 到达终点
```

### 134. 加油站

#### 核心思路

将每站净收益记为 `gas[i] - cost[i]`，问题转化为：找起点使得从该点出发的前缀和始终 >= 0，且整圈总和 >= 0。

两个关键变量：
- `total`：所有站净收益之和。若 `total < 0`，无论从哪出发都不够用，返回 -1。
- `tank`：从 `start` 出发的累积净收益。若 `tank < 0`，说明 `[start, i]` 内任意位置出发都不够用，重置 `start = i+1, tank = 0`。

**为什么可以跳过整段**：若从区间 `[start, k]`（k <= i）中的某点出发，`[start, k-1]` 段累积 >= 0（否则会更早被排除），加上 `[k, i]` 段（已知此段和为负），总和仍 < 0——整段都不行，直接跳到 `i+1`。

#### Python 代码

```python
def canCompleteCircuit(gas: list[int], cost: list[int]) -> int:
    total = 0   # 全局总净收益，判断有无解
    tank = 0    # 从 start 出发的累积净收益
    start = 0   # 当前候选起点

    for i in range(len(gas)):
        diff = gas[i] - cost[i]
        total += diff
        tank += diff
        if tank < 0:            # [start, i] 区间任意起点都不行
            start = i + 1
            tank = 0

    return start if total >= 0 else -1
```

#### 图解

```text
gas  = [1, 2, 3, 4, 5]
cost = [3, 4, 5, 1, 2]
diff = [-2,-2,-2, 3, 3]

i=0: tank = -2 < 0 → start=1, tank=0
i=1: tank = -2 < 0 → start=2, tank=0
i=2: tank = -2 < 0 → start=3, tank=0
i=3: tank = 0+3 = 3
i=4: tank = 3+3 = 6

total = -2-2-2+3+3 = 0 >= 0 → 返回 start=3 ✓
（从下标 3 出发：+3, +3, -2, -2, -2，累积差始终 >= 0）
```

### 135. 分发糖果

#### 核心思路

每个孩子同时受到两个约束：
1. 若 `ratings[i] > ratings[i-1]`，则 `candy[i] > candy[i-1]`（左邻约束）。
2. 若 `ratings[i] > ratings[i+1]`，则 `candy[i] > candy[i+1]`（右邻约束）。

两个方向的约束独立处理，分两次扫描：
- **从左到右**：只满足「右边比左边评分高则糖果更多」。
- **从右到左**：只满足「左边比右边评分高则糖果更多」，并取 `max` 保留左扫结果。

**为什么第二次必须从右向左**：位置 `i` 的糖果数依赖 `candy[i+1]` 的最终值，而 `candy[i+1]` 是在处理更右侧位置时才确定的；从右向左遍历保证 `candy[i+1]` 已经过右侧约束修正，从左向右则拿到的是未修正的旧值。

#### Python 代码

```python
def candy(ratings: list[int]) -> int:
    n = len(ratings)
    candy = [1] * n              # 每人初始分配 1 颗

    # 第一次：从左到右，满足「右高于左则多一颗」
    for i in range(1, n):
        if ratings[i] > ratings[i - 1]:
            candy[i] = candy[i - 1] + 1

    # 第二次：从右到左，满足「左高于右则多一颗」
    for i in range(n - 2, -1, -1):
        if ratings[i] > ratings[i + 1]:
            candy[i] = max(candy[i], candy[i + 1] + 1)

    return sum(candy)
```

#### 图解

```text
ratings = [1, 0, 2]

初始 candy: [1, 1, 1]

从左到右：
  i=1: 0 < 1，不变         candy: [1, 1, 1]
  i=2: 2 > 0，+1           candy: [1, 1, 2]

从右到左：
  i=1: 0 < 2，不变         candy: [1, 1, 2]
  i=0: 1 > 0 → max(1, 1+1)=2   candy: [2, 1, 2]

sum = 5 ✓

ratings = [1, 2, 2]

初始 candy: [1, 1, 1]

从左到右：
  i=1: 2 > 1 → candy[1]=2  candy: [1, 2, 1]
  i=2: 2 == 2，不变         candy: [1, 2, 1]

从右到左：
  i=1: 2 == 2，不变         candy: [1, 2, 1]
  i=0: 1 < 2，不变          candy: [1, 2, 1]

sum = 4 ✓（相邻评分相等时两人糖果数可以相同）
```

### 860. 柠檬水找零

#### 核心思路

柠檬水售价 5 美元，顾客支付 5、10 或 20 美元，需要正确找零。

三种情况：
- **收到 5**：直接收，`five += 1`，不需找零。
- **收到 10**：找 5 元，`five -= 1, ten += 1`；若无 5 元则失败。
- **收到 20**：需找 15 元，优先用 `10+5`（消耗 1 张 5 元）；没有 10 元时用 `5×3`（消耗 3 张 5 元）；均不够则失败。

**贪心证明**：5 元可用于两种找零场景（找 10 元账单、凑 20 元账单），是「万能零钱」；10 元只能用于凑 20 元账单。收到 20 时优先消耗 `10+5` 比 `5+5+5` 少消耗 2 张 5 元，为后续 10 元账单的找零保留了更多 5 元——局部消耗少的选择对后续全局更有利。

#### Python 代码

```python
def lemonadeChange(bills: list[int]) -> bool:
    five = 0    # 手头 5 元数量
    ten = 0     # 手头 10 元数量
    # 20 元不需要记录——找零绝不使用 20 元

    for bill in bills:
        if bill == 5:
            five += 1
        elif bill == 10:
            if five == 0:
                return False        # 无法找 5 元零钱
            five -= 1
            ten += 1
        else:                       # bill == 20，需找 15 元
            if ten > 0 and five > 0:
                ten -= 1            # 贪心：优先用 10+5，保留 5 元备用
                five -= 1
            elif five >= 3:
                five -= 3           # 退而求其次，用三张 5 元
            else:
                return False        # 无法凑出 15 元找零
    return True
```

#### 图解

```text
bills = [5, 5, 5, 10, 20]

收  5: five=1, ten=0
收  5: five=2, ten=0
收  5: five=3, ten=0
收 10: 找 5 元 → five=2, ten=1
收 20: 优先 10+5 → ten=0, five=1 → 返回 True ✓

bills = [5, 5, 10, 10, 20]

收  5: five=1
收  5: five=2
收 10: five=1, ten=1
收 10: five=0, ten=2
收 20: ten>0 但 five=0，不能用 10+5；
       five<3，不能用 5×3 → 返回 False ✓
```

## 统一对比

| 题号 | 题目 | 核心变量 | 贪心策略 | 时间 | 空间 |
|------|------|----------|----------|------|------|
| 55 | 跳跃游戏 | `max_reach` | 维护最远可达，超出则不可达 | O(n) | O(1) |
| 45 | 跳跃游戏 II | `cur_end`, `next_end` | 区间接力，边界处强制跳 | O(n) | O(1) |
| 134 | 加油站 | `total`, `tank`, `start` | 累积为负则重置起点 | O(n) | O(1) |
| 135 | 分发糖果 | `candy[]` | 两次单向扫描，取 max | O(n) | O(n) |
| 860 | 柠檬水找零 | `five`, `ten` | 优先消耗最不万能的零钱 | O(n) | O(1) |

**分类视角**：

- **覆盖范围模型**：55 和 45 是同一「最远可达」模型的两个层次——55 问能否到达，45 问最少几步。
- **优先保留通用资源**：134 保留最优起点（不断右移排除不可行区间）；860 保留 5 元（优先消耗 10 元）。
- **多维约束拆分**：135 的左右约束互不干扰，分向处理后取 max 合并，是「约束解耦」思维的典型示例。

  两次扫描莫混搭，左到右保右高多，
  右到左取 max 保左高，两侧约束同时到。

  10 用完才三张 5，顺序颠倒必出错。

## 总结与扩展

### 五题的共同规律

这五道题共同体现了贪心的本质——**不穷举全局，只在每步做局部最优选择，并验证局部最优可以推出全局最优**：

- **55**：不枚举跳法，只维护「最远能到哪」，局部贪心（每步取最远）积累为全局可达性。
- **45**：不枚举路径，只维护「当前段边界与下一段上界」，按段贪心（每段尽量远）得到最少跳跃。
- **134**：不枚举起点，利用「累积差为负则排除整段」的剪枝，O(n) 一次确定唯一解。
- **135**：不同时考虑两侧约束，拆分为两次单向扫描，分治贪心，取 max 同时满足两侧。
- **860**：不穷举找零方案，只优先消耗「用途最窄的零钱」，选择贪心保留最大灵活性。

### 扩展方向

- **55 的变体**：LeetCode 1306（跳跃游戏 III）和 1345（跳跃游戏 IV）增加了双向跳跃或图结构，`max_reach` 不再适用，需改用 BFS。
- **45 的变体**：LeetCode 1871（跳跃游戏 VII）引入可跳跃区间，需要前缀和优化的贪心或滑动窗口。
- **134 的变体**：LeetCode 871（最低加油次数）改为优先队列贪心——每次「尽量加最多的油」，需最大堆辅助，时间复杂度 O(n log n)。
- **135 的延伸**：「两次扫描 + 取 max」思路可推广至任何「左右两侧约束分别独立优化」的场景，是处理相邻约束冲突的通用范式。
- **860 的延伸**：任何「有限面值找零」或「有限资源分配」问题，都可以思考「哪种资源最通用，哪种最局限」，优先消耗局限的那种——这是 860 贪心策略的本质。
